# Repository Engineering Guide

## Purpose

This repository is an Android-only Flutter plugin project with an example
Android application for a native image downloading and caching assignment.
Changes must prioritize correctness at the native/Flutter boundary, observable
cache semantics, clean dependency direction, and reviewability.

The repository root is the `image_cache_plugin` Flutter package. The standard
`example/` directory is the demonstration application. Do not add a redundant
nested plugin directory.

## Local Planning Files

`PLAN.md` and `PLAN_IMPL.md` are local working documents. Keep both updated as
architecture and implementation status change, but never commit them and never
add them to `.gitignore`. Before staging any change, explicitly verify that
neither file is included.

Keep this `AGENTS.md` synchronized whenever actual paths, commands,
dependencies, architecture rules, or delivery workflow change.

## Architecture Boundaries

### Native Android

- `domain` owns contracts, result types, and typed errors.
- `data` owns `HttpURLConnection`, files, hashing, cache implementation, and
  repository implementation.
- `api` owns the public `ImageLoader` facade for Java and Kotlin hosts.
- `bridge` is the only package allowed to import Flutter embedding classes.
- Native domain, data, and API code must remain usable without Flutter types.
- Keep blocking network, file, and bitmap work off the main thread.
- Make dispatchers and time injectable where deterministic tests need control.
- Own every coroutine scope explicitly and cancel it with the relevant lifecycle.
- Never use `GlobalScope`.

### Dart plugin

- Keep the public API small and documented.
- Keep `flutter_bloc`, `fpdart`, `get_it`, and `equatable` out of the plugin's
  public API and runtime dependencies.
- Use a client abstraction for channel operations so widgets can be tested.
- Validate dynamic MethodChannel results before constructing typed models.
- Do not transfer full encoded image bytes over MethodChannel.
- Render native files through a file-backed image provider.
- Do not start Futures from a widget's `build` method.
- Handle URL changes, disposal, and stale asynchronous completions.

### Example app

- Organize the gallery feature into data, domain, and presentation layers.
- Dependencies point inward: presentation to domain, data to domain.
- Domain must not import Flutter widgets, `dart:io`, MethodChannel, or get_it.
- Data sources handle external systems and may throw typed data exceptions.
- Repositories map expected exceptions into `fpdart` `Either<Failure, T>`.
- Use cases expose business operations to presentation.
- Cubits are the MVVM ViewModels.
- Views render state and forward user intent; they do not fetch or parse data.
- Use `get_it` only in the composition root. Inject dependencies through
  constructors everywhere else.
- Register Cubits as factories and long-lived stateless services as lazy
  singletons.
- Use Equatable for entities, failures, and Cubit states.

## Runtime Dependency Rules

The native image downloader/cache must not use Glide, Coil, Picasso, OkHttp,
Retrofit, Room, or another third-party image/network/persistence SDK.

Allowed plugin runtime dependencies:

- Flutter SDK.
- Kotlin standard library.
- Kotlin Coroutines, required by the assignment.

Allowed example architecture dependencies:

- `flutter_bloc` 9.1.1.
- `fpdart`.
- `get_it`.
- `equatable`.

Before adding any dependency, document the need, scope, and why a platform or
existing dependency cannot satisfy it. Do not add code generation solely to
avoid writing a small explicit model or registration graph.

## Native Cache Invariants

All implementation and review work must preserve these guarantees:

- A committed entry is either complete and valid or absent.
- Freshness is four hours from successful commit.
- Expired files are never returned as valid cache hits.
- Files survive ordinary process death and app relaunch.
- Same-URL concurrent requests perform at most one network download.
- Different URLs may download concurrently.
- Multiple loader/repository instances in the same application process obey
  the same synchronization and invalidation guarantees.
- URL eviction cannot be undone by an older request.
- Full clear cannot be undone by older requests.
- Failed and cancelled operations do not leave valid entries.
- A network refresh returns a new file path to avoid stale Flutter `FileImage`
  identity.
- Streams, file handles, and connections close on success, error, and
  cancellation.
- A committed filename encodes the successful commit timestamp and unique
  identity. Allocate that timestamp immediately before atomic rename while
  holding the mutation mutex, and return it as `cachedAtMilliseconds`.
- If crash recovery finds multiple committed files, the newest valid identity
  is authoritative and older committed orphans are removed under coordination.
- Images with non-positive or configured excessive width, height, or pixel
  count are rejected before commit.

Hash the validated URL string exactly with SHA-256; do not perform unsafe URL
normalization. Use same-directory temporary files and timestamped unique
committed names. A process-wide coordinator keyed by canonical cache-root path
owns URL locks, generations, active temporary identities, and a mutation mutex.
Generation validation and atomic rename must occur in the same critical section
used by eviction, clear, and pruning. Pruning must skip active temporary files
and revalidate committed identity before deletion. Cross-process coordination
is not required, but the limitation must be documented. Do not weaken an
invariant for convenience.

## Platform Channel Contract

Supported methods:

- `loadImage` with URL string.
- `evictImage` with URL string.
- `clearCache` without arguments.

Successful `loadImage` returns a map containing:

- `path`: non-empty String.
- `source`: `network` or `disk`.
- `cachedAtMilliseconds`: integer epoch milliseconds.

Stable error codes:

- `invalid_argument`.
- `network_error`.
- `http_error`.
- `cache_error`.
- `invalid_image`.
- `cancelled`.
- `internal_error`.

Every native method call must complete its result exactly once. Channel calls
arrive on the main thread; native work may switch to IO but result delivery must
return to the main thread. Track pending calls with atomic completion guards.
On detach, stop accepting calls, settle pending calls with `cancelled` while the
binding is valid, then cancel jobs and remove the handler. Ignore late
completions.

## Public Native API

The native facade must be ergonomic from both Kotlin and Java:

- Kotlin callers receive suspend APIs.
- Java callers receive callback APIs without direct Continuation handling.
- Native targets use `ImageView` and Drawable/resource placeholders.
- Target updates and callbacks occur on the main thread.
- Reused targets cancel or supersede earlier requests.
- Public nullability and error behavior are explicit.

Any public API change requires corresponding Kotlin example, Java example,
compile check, tests, and README update.

## Error Handling

- Use typed native exceptions internally.
- Map native types to stable MethodChannel error codes.
- Parse channel errors into typed Dart exceptions.
- Map expected example data exceptions into domain Failures.
- Return expected domain failures in `Either`; do not throw them into Cubits.
- Keep raw stack traces and internal paths out of user-visible messages.
- Do not catch `Throwable` broadly in a way that swallows cancellation.
- Preserve cancellation semantics through coroutine layers.

## Testing Standards

Automated tests are not an assignment requirement. Keep the foundation smoke
tests and add only a concise, high-signal set that demonstrates senior testing
skill around critical correctness, boundaries, and state transitions. Avoid
exhaustive matrices, repetitive delegation tests, and one test per error code,
malformed DTO field, redirect permutation, or UI state. Focused tests should
ship in the same PR as the behavior they cover.

### Android

- Use Mockito-Kotlin for collaborator mocks.
- Use Coroutines Test for deterministic scheduling and virtual time.
- Use JUnit Jupiter 5.13.4 on JUnit Platform 1.13.4.
- Use real temporary directories for disk behavior.
- Preserve tests for fresh/expired persistence, reopening a fresh entry,
  same-URL coalescing, invalidation that older work cannot undo, and failures
  leaving no valid entry.
- Use only a small representative downloader and image-validation sample.
- Add Java compile source for public Java interoperability.
- Avoid repetitive native facade/API delegation tests.

### Dart and Flutter

- Use Mocktail for mocked boundaries.
- Use `bloc_test` 10.0.0 for Cubit transition tests.
- Use Flutter's test binary messenger for actual MethodChannel contract tests.
- Test core Dart result parsing with representative malformed/error cases rather
  than every field and code.
- Cover core plugin widget placeholder, success, failure, and stale-result
  behavior with a small focused set.
- Test repository `Either` mapping and key Cubit load success/failure and clear
  success/failure transitions.
- Keep one representative gallery widget test rather than a UI-state matrix.
- Reset get_it between tests; tests must not depend on execution order.
- Avoid live network access in unit and widget tests.

### Bridge lifecycle

- Keep a small contract/lifecycle sample covering representative completion,
  main-thread delivery, and detach with a pending call and late completion.
- Do not repeat lifecycle scenarios for every channel method.

### Integration

- One device integration flow is optional and may use the supplied endpoint.
- Keep live network integration outside the fast mandatory unit-test job.
- Document network and device prerequisites.
- If included, cover network source, disk source, and one invalidation operation.

Do not write handwritten fake implementations when Mockito, Mocktail, or the
real lightweight value/storage object provides a clearer test.

## Code Style

- Prefer the smallest correct design and avoid speculative abstractions.
- Use descriptive domain names, not generic `Manager`, `Helper`, or `Utils`.
- Keep one class per responsibility, but do not split trivial logic into layers
  that add no boundary or test value.
- Prefer immutable data.
- Keep comments rare and focused on non-obvious invariants or tradeoffs.
- Public Dart and native APIs require documentation.
- Keep code and documentation ASCII unless a user-facing requirement needs
  Unicode.
- Format Dart and Kotlin code with project-standard tools.
- Do not leave template counters, platform-version methods, TODOs, or dead code.

## Git Workflow

The repository starts with one direct bootstrap commit on `main`. After that,
all work goes through milestone branches and PRs:

1. `chore/project-foundation`
2. `feat/native-image-loader`
3. `feat/flutter-plugin-api`
4. `feat/example-gallery-domain`
5. `feat/example-gallery-ui`
6. `chore/delivery-hardening`

Use Conventional Commits with meaningful scopes where useful. Keep commits
reviewable and avoid mixing unrelated cleanup into feature commits.

Before committing:

1. Inspect `git status --short --branch`.
2. Inspect `git diff` and `git diff --staged`.
3. Inspect recent `git log --oneline -10` after history exists.
4. Run relevant tests and checks.
5. Stage only intended files.
6. Confirm `PLAN.md` and `PLAN_IMPL.md` are not staged.
7. Check for credentials, local paths, signing files, and generated output.

Do not amend, force-push, skip hooks, or rewrite shared history unless the user
explicitly requests it. Never merge a PR on the user's behalf. After opening a
milestone PR, report its URL and verification, then wait for the user to review
and merge it before starting the next branch.

## Completion Checklist

A change is complete only when:

- It respects dependency direction and cache invariants.
- Its error and cancellation paths are handled.
- Important behavior has automated tests using the agreed test tools.
- Dart format, analysis, relevant Flutter tests, relevant Android tests, and
  required builds pass.
- Public APIs and README are updated when needed.
- `PLAN.md`, `PLAN_IMPL.md`, and this guide reflect any changed decision.
- The diff contains no unrelated or local-only files.
